# AGENTS.md

这个 vault 是一个由 Codex 协助维护的持续型学习 wiki。

## 目标

这个 vault 主要用于：

- 学习 Node.js 后端开发
- 学习 AI、LLM 与相关工程知识
- 将零散原始资料沉淀为结构化、可复用的知识库

人负责选择资料、提出问题、决定优先级。
Agent 负责阅读、总结、交叉链接、更新页面，并尽量保持整个 wiki 一致。

## 三层结构

这个 vault 分为三层：

1. `raw/`
   原始资料层。这里的内容视为不可变 source documents。
2. `wiki/`
   结构化知识层。这里的 markdown 页面由 agent 维护。
3. `learning-records/`
   学习记录层。这里放当前目标、进度、复盘和反思。

不要在没有说明的情况下改写 source 的原意。
如果有不确定、冲突或版本变化，要明确标注。
优先把有长期价值的内容写回 wiki，而不是只停留在一次性对话里。

## 目录说明

- `raw/inbox/` 放新抓取、尚未处理的资料
- `raw/sources/` 放整理后保留的来源记录
- `wiki/overviews/` 放主题地图和高层总结
- `wiki/concepts/` 放概念页
- `wiki/projects/` 放通用的实战模式和实现思路
- `wiki/sources/` 放单篇来源总结
- `learning-records/` 放当前重点、学习日志和复盘
- `projects/` 放代码练习或项目实践
- `templates/` 放可复用模板

## 默认工作流

### Ingest

当加入新的学习资料时：

1. 从 `raw/inbox/` 或其他 raw 位置读取原始资料。
2. 在 `wiki/sources/` 中创建该资料的总结页。
3. 更新相关的 `wiki/concepts/` 概念页。
4. 如果资料改变了整体认识，至少同步更新一个 `wiki/overviews/` 页面。
5. 更新 `index.md`。
6. 在 `log.md` 里追加一条简短记录。
7. 如果资料改变了当前阶段重点，更新 `learning-records/current-focus.md`。

### Query

当回答学习问题时：

1. 先读 `index.md`。
2. 再读最相关的 wiki 页面。
3. 优先引用或链接 vault 里已有页面。
4. 如果这次回答产生了长期有价值的理解，把它写回 wiki，不要只留在聊天记录里。

### Lint

定期检查：

- 孤儿页面或几乎没有链接的页面
- 重复的概念页
- 已被新资料推翻或更新的旧结论
- 在多处出现但还没有单独页面的重要概念
- 本应互相链接却尚未建立连接的页面

有意义的维护动作要记录到 `log.md`。

## 编辑规则

- 优先使用 markdown 和 Obsidian wikilink，例如 `[[index]]`
- 页面尽量简洁，并保持互相链接
- 尽量一页只讲一个主题
- 长期页面末尾加上简短的“相关”部分
- 除非用户明确要求，不要覆盖用户亲自写下的内容
- 不要静默删除重要旧结论；如果结论更新了，要说明为什么更新

## 命名规则

- 持久页面文件名使用小写 kebab-case
- source summary 文件名优先使用 `YYYY-MM-DD-short-title.md`
- 如果资料没有明确日期，可以使用 `source-short-title.md`

## 当前学习重点

当前优先方向：

1. Node.js 后端基础
2. 实用后端项目与实现模式
3. AI 与 LLM 工程概念
4. 在 Obsidian 中建设可持续维护的个人知识库

如果需要取舍，优先选择更能帮助理解的产物：

- 例子
- 对比
- 小练习
- 面向项目的学习输出

## 偏好的输出方式

生成或更新笔记时：

- 先给出通俗总结
- 区分事实与解释
- 合适时补充例子
- 如果主题还没讲透，加入“下一步问题”或“待继续追问”
- 保证页面脱离当前聊天后依然有复习价值
